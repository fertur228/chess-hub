- Language: TypeScript with strict mode enabled.
- Runtime/UI: React 19.
- Styling: Tailwind CSS v4, shadcn/ui, Radix UI primitives, lucide-react icons.
- Routing: TanStack Router with file-based routes and generated route tree.

#### Phase 5B (2026-05-04)
- **Security**: Added `forfeit-game` edge function. Resignation and abandonment logic now routes through server validation.
- **TanStack Router**: Navigation blockers implemented successfully utilizing a synchronous `user` check. A reference `allowNavigationRef` fall-through successfully hardens the game abandon trap without infinite blocking loops.

#### Phase 5C (2026-05-04)
- **Security**: Permanent decommissioning of direct client `UPDATE` SQL generation against the `rooms` dataset. A unified wrapper via Supabase Functions establishes total deterministic trust verification against in-game movements and forfeits. Only `INSERT` operations implicitly tied to new room starts and standard `SELECT` read paths remain directly addressable.

- **Phase 5C verified**: RLS lockdown proved successful. No regressions observed across matchmaking or secure RPCs.
- **Deferred Risks**: Heartbeat presence (preventing zombie ghost games from rage-quits) and server-side ratings computation remain open until Phase 6+.

#### Phase 6A Blueprint Analysis
- **Security Gap Identified**: Client executes `profiles.update(rating)` post-match organically.
- **Resolution Blueprint**: Centralizing ratings natively onto a `finalize_online_room` PostgreSQL RPC guarantees atomic calculations locked away from client manipulation. Incorporating a UNIQUE `source_room_id` ensures absolute mathematical idempotency bypassing Edge Function race conditions.

- **Phase 6B verification confirmed**: Security infrastructure dynamically locks rating calculations globally without logic gaps natively.

#### Phase 6C Audit
- Profile table explicit edit policy dropped natively preventing explicit arbitrary updates dynamically locking down ranks & stat limits explicitly securely.

#### Phase 6D Audit
- **Security**: Ranked private rooms are now blocked at both UI and database levels. Private rooms are casual-only to reduce coordinated rating abuse.
- **Matchmaking**: `find_or_create_public_room` remains `SECURITY DEFINER` with `SET search_path = public`, uses `auth.uid()`, and matches public rooms only by requested mode.
- **Ranked queue**: Ranked matching loads the caller's current profile rating and prioritizes opponents within +/-200, then +/-400, then any waiting ranked host, using `FOR UPDATE SKIP LOCKED`.
- **Remaining risk**: Public ranked matchmaking can still pair friends if they queue at the same time and no closer opponent exists. Further mitigation could add cooldowns, repeat-opponent limits, or suspicious-pattern review.

#### Phase 6D Ranked Forfeit Verification
- **Verified**: Ranked explicit Resign and ranked Leave and resign both finalize as rated losses for the leaver and rated wins for the opponent.
- **Verified**: Each ranked forfeit creates exactly one `games` row with `source_room_id` and two `rating_events` rows.
- **Verified**: Reopening or refreshing the finished room does not reapply rating.
- **Verified**: Casual resign/leave updates counters while preserving ratings.
- **Verified**: Leaderboard reflects ranked rating changes after forfeit finalization.
- **Pre-deploy recommendation**: Run one focused gameplay integrity sweep before Cloudflare deployment, especially ranked/casual finalization, reconnect/reopen behavior, profile refresh, history, review links, and leaderboard consistency.

#### Pre-deploy Record Move Finalization Audit
- **Finding**: `record-move` could set `rooms.status = 'finished'` for checkmate/stalemate/draw and call `finalize_online_room` without handling `finalizeError`.
- **Risk**: A terminal ranked move could appear successful to clients with `game_id = null` if game save/rating finalization failed.
- **Fix**: `record-move` now mirrors `forfeit-game` finalization error handling and returns `GAME_FINALIZATION_FAILED` with a clear message.
- **Deployment required**: `npx supabase functions deploy record-move`.

#### Phase 7A (Cloudflare deployment readiness)
- **Platform**: Deploy as **Cloudflare Workers** full-stack SSR, not Pages-only hosting. **`vite build`** outputs **`dist/client`** (asset binding) plus **`dist/server`** (Worker **`index.js`** and generated **`dist/server/wrangler.json`**).
- **Root config**: `wrangler.jsonc` uses `main` = `@tanstack/react-start/server-entry`, `nodejs_compat`, and **`observability.enabled`**. Operators apply deploy with **`npm run deploy`** or **`wrangler deploy --config dist/server/wrangler.json`** after a successful **`npm run build`**.
- **Environment**: **`VITE_SUPABASE_*`** must be present at **`npm run build`** time so the client bundle resolves Supabase. Worker runtime expects **`SUPABASE_URL`** + **`SUPABASE_PUBLISHABLE_KEY`**. **`SUPABASE_SERVICE_ROLE_KEY`** is documented for optional **`supabaseAdmin`** usage only; unused by current routes. Never ship service role keys as **`VITE_*`** values.
- **Verify**: **`npm.cmd run build`** passes; **`wrangler deploy … --dry-run`** passes after build.
